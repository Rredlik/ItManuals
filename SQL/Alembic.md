Похоже, ваша база данных уже содержит таблицы (например, "Users"), которые первая миграция (`70b74caa6503_Initial_revision.py`) пытается создать заново. Это частая ситуация при настройке Alembic для существующей базы данных.

Alembic не знает, что эти таблицы уже есть и соответствуют (или должны соответствовать) начальному состоянию ваших моделей.

**Вот как это можно исправить:**

1. **Убедитесь, что ваши модели (`db/models.py`) сейчас в актуальном состоянии:** То есть, они должны включать колонки `amount_expected` и `amount_paid`, так как вы хотите в итоге прийти к этому состоянию.
    
2. **Сделайте первую миграцию "пустой" по отношению к существующим таблицам:**
    
    - Откройте файл первой миграции: `migrations/versions/70b74caa6503_initial_revision.py` (или тот, который у вас был первым и вызвал ошибку).
    - **Если этот скрипт пытается создать таблицы, которые у вас уже есть (`op.create_table('Users', ...)` и т.д.):**
        - Удалите или закомментируйте все команды `op.create_table(...)` внутри функции `upgrade()` для тех таблиц, которые уже существуют в вашей БД.
        - Аналогично, удалите или закомментируйте соответствующие `op.drop_table(...)` в функции `downgrade()`.
        - Если все таблицы, которые он пытается создать, уже существуют, то функции `upgrade()` и `downgrade()` в этом файле могут просто содержать `pass`.
        - **Цель:** Этот первый скрипт не должен пытаться создавать таблицы, которые уже есть. Он должен отражать тот факт, что "база уже содержит эти таблицы в их первоначальном виде (до добавления новых колонок)".
3. **"Отметьте" базу данных этой "пустой" или скорректированной первой миграцией:**
    
    - Выполните команду (замените `70b74caa6503` на актуальный ID вашей первой миграции, если он другой):
        
```
alembic stamp 70b74caa6503
```
        
    - Эта команда скажет Alembic: "Считай, что база данных уже находится в состоянии, описанном миграцией `70b74caa6503`". Она не будет выполнять `upgrade()` из этого скрипта, а просто запишет этот номер ревизии в таблицу `alembic_version`.

4. **Теперь примените следующую миграцию (которая добавляет колонки):**
    
    - Теперь, когда Alembic знает "базовое" состояние, вы можете применить миграцию, которая добавляет новые колонки (`amount_expected`, `amount_paid`). Если у вас есть вторая миграция (например, `def456..._add_amount_columns_to_payment_tables.py`), которая делает это, выполните:
        
```
alembic upgrade head
```
        
        Или, если вы хотите применить только следующую за `70b74caa6503`:
        
```
alembic upgrade <ID_второй_миграции>
```
        
    - Это должно добавить недостающие колонки `amount_expected` и `amount_paid` к вашим существующим таблицам `payments` и `repeat_payments`.

5. **Проверка и генерация новых миграций:**
    
    - После этого, если вы снова запустите `alembic revision -m "новое_изменение" --autogenerate` (при условии, что ваши модели `db/models.py` актуальны), ошибка "Target database is not up to date" должна исчезнуть. Если нет новых изменений в моделях по сравнению с тем, что теперь есть в БД, Alembic должен сгенерировать пустую миграцию.

**Кратко:** Проблема в том, что ваша первая миграция пытается создать таблицы, которые уже существуют. Вам нужно либо отредактировать эту первую миграцию, чтобы она не создавала существующие таблицы, либо, если схема базы данных уже полностью соответствует вашим моделям (включая новые колонки), просто "отметить" базу последней версией миграции (`alembic stamp head`). Судя по ошибке, первый вариант (корректировка первой миграции) более правильный.
